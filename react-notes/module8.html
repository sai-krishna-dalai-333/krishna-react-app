Server State Management (React Query)

React Masterclass Module 8

← Back to Roadmap

📝 How to use this guide: Historically, developers used Redux to store data fetched from APIs. Today, the industry has shifted to separating "Client State" (UI toggles) from "Server State" (API data). TanStack Query (React Query) is the undisputed king of Server State. Interviewers will highly value your understanding of caching, background fetching, and query invalidation.

1. What is React Query & Why do we need it?

1. Definition

React Query (now TanStack Query) is a powerful data-fetching and state management library that specifically handles asynchronous operations, caching, background updates, and stale data synchronization between the server and the client.

2. Explanation

  • The Problem: Using `useEffect` and `useState` to fetch data requires manual tracking of `isLoading`, `isError`, and `data` states. Furthermore, `useEffect` fetches data every time a component mounts, leading to redundant network requests.
  • The Solution: React Query caches the API response. If you navigate to another page and come back, it instantly shows the cached data while secretly refetching in the background to ensure it's up-to-date.
  • It cleanly separates Server State (data owned by the database) from Client State (data owned by the UI, like a dark mode toggle).

3. Example Code

// The Old Way (Boilerplate Hell)
const [data, setData] = useState(null);
const [isLoading, setIsLoading] = useState(true);
const [error, setError] = useState(null);
useEffect(() => {
  fetchData().then(setData).catch(setError).finally(() => setIsLoading(false));
}, []);

// The React Query Way
const { data, isLoading, error } = useQuery({ 
  queryKey: ['users'], 
  queryFn: fetchUsers 
});

4. Real-Time Scenarios & Usages

  • Any application that communicates with a REST or GraphQL API.
  • Dashboards where multiple components on the same page need the exact same API data (React Query automatically deduplicates the requests into a single network call).

5. Advantages & Disadvantages

Advantages: Eliminates hundreds of lines of boilerplate code. Provides aggressive caching, automatic retries on failure, and window-focus refetching out of the box.
Disadvantages: Caching invalidation logic can get complex in highly mutating applications. Adds to bundle size.

⚡ Interview Corner

Q: Why is it an anti-pattern to store API response data in Redux nowadays?
A: Server data is inherently out-of-sync the moment it hits the client. Redux requires you to manually write actions and reducers to fetch, store, and update this data, but it offers no built-in caching or background refetching mechanics. React Query is designed explicitly for the asynchronous, shared nature of Server State, leaving Redux to handle purely synchronous UI state.

2. Setting up QueryClient and QueryClientProvider

1. Definition

The `QueryClient` is the core engine that manages queries, caching, and state. The `QueryClientProvider` is a Context provider that makes this engine accessible to your entire React component tree.

2. Explanation

  • You must instantiate a new QueryClient() outside of your App component (or wrap it in a `useState` / `useRef` if inside) so it doesn't get recreated on every render.
  • You can pass default options to the QueryClient to configure global behaviors, such as how long data stays fresh (staleTime) or how many times it should retry failed requests.

3. Example Code

import { QueryClient, QueryClientProvider } from '@tanstack/react-query';
import { ReactQueryDevtools } from '@tanstack/react-query-devtools'; // Bonus!

// 1. Create a client with global defaults
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 1000 * 60 * 5, // Data is "fresh" for 5 minutes
      retry: 2, // Retry failed requests twice before showing an error
      refetchOnWindowFocus: false, // Don't refetch when user switches browser tabs
    },
  },
});

const App = () => {
  // 2. Wrap your application
  return (
    <QueryClientProvider client={queryClient}>
      <MainApplication />
      <ReactQueryDevtools initialIsOpen={false} /> {/* Visual cache debugger */}
    </QueryClientProvider>
  );
};

4. Real-Time Scenarios & Usages

  • Configuring global retry logic so a minor network blip doesn't instantly show a red error screen to the user.
  • Wrapping the root index.js or main.jsx file to enable React Query across all routes.

5. Advantages & Disadvantages

Advantages: Centralizes cache configuration. The React Query Devtools are legendary for debugging API states visually.
Disadvantages: Global defaults might not suit every single endpoint, requiring you to carefully override settings on specific queries.

⚡ Interview Corner

Q: Where is the React Query cache actually stored? Is it in localStorage?
A: By default, the cache is stored strictly in JavaScript memory (RAM) within the QueryClient instance. It does not persist in localStorage across hard browser refreshes (though TanStack provides an experimental persister plugin if you explicitly want that behavior).

3. useQuery: Fetching & Caching

1. Definition

`useQuery` is the primary hook used to fetch and cache data from a server. It requires a unique `queryKey` and an asynchronous `queryFn`.

2. Explanation

  • queryKey: An array (e.g., ['users', userId]) that uniquely identifies the cache entry. If the variables in the key change, React Query automatically triggers a new fetch.
  • queryFn: The actual asynchronous function (using Axios or Fetch) that returns a Promise resolving to data.
  • staleTime vs gcTime (cacheTime): staleTime is how long data is considered "fresh" before a background refetch is triggered. gcTime is how long inactive/unmounted data stays in RAM before being garbage collected and deleted completely.

3. Example Code

import { useQuery } from '@tanstack/react-query';
import axios from 'axios';

const fetchUserById = async (id) => {
  const { data } = await axios.get(`/api/users/${id}`);
  return data;
};

const UserProfile = ({ userId }) => {
  const { data: user, isLoading, isError, error } = useQuery({
    queryKey: ['user', userId], // Key depends on the ID!
    queryFn: () => fetchUserById(userId),
    staleTime: 60000, // Data remains fresh for 1 minute
  });

  if (isLoading) return <Spinner />;
  if (isError) return <p>Error: {error.message}</p>;

  return <h1>{user.name}</h1>;
};

4. Real-Time Scenarios & Usages

  • Fetching paginated data where the queryKey looks like ['posts', { page: 2 }].
  • Automatically deduping requests: If ComponentA and ComponentB both call useQuery(['settings']) at the exact same time, only ONE network request is actually sent.

5. Advantages & Disadvantages

Advantages: Exposes exact UI states (isLoading vs isFetching). Drastically speeds up perceived app performance via caching.
Disadvantages: Query keys must be meticulously organized. If you misspell a query key, caching and invalidation will fail silently.

⚡ Interview Corner

Q: What is the exact difference between `isLoading` and `isFetching` in React Query?
A: isLoading is true ONLY when the query is fetching for the very first time and there is absolutely no cached data to display. isFetching is true ANYTIME a background network request is currently in flight. If you have cached data, navigating back to a page will show the UI instantly (isLoading: false), but isFetching will be true while it silently updates the cache in the background.

4. useMutation: Creating & Updating Data

1. Definition

While `useQuery` is for reading data (GET), `useMutation` is used for creating, updating, or deleting data (POST, PUT, DELETE). It handles the lifecycle of an action that modifies server state.

2. Explanation

  • Unlike useQuery, mutations do NOT run automatically on render. They return a mutate function that you must trigger manually (e.g., on a button click).
  • Mutations don't use queryKeys for caching, because you generally don't cache the result of a POST request.
  • It provides highly useful callback hooks: onMutate (fires before request), onSuccess, onError, and onSettled.

3. Example Code

import { useMutation } from '@tanstack/react-query';
import axios from 'axios';

const CreatePostForm = () => {
  const mutation = useMutation({
    mutationFn: (newPost) => axios.post('/api/posts', newPost),
    onSuccess: (data) => {
      // Fires if the network request succeeds
      alert("Post created successfully!");
    },
    onError: (error) => {
      alert("Failed to create post.");
    }
  });

  const handleSubmit = (e) => {
    e.preventDefault();
    // Trigger the mutation manually, passing the variables
    mutation.mutate({ title: 'New Post', content: 'Hello World' });
  };

  return (
    <form onSubmit={handleSubmit}>
      <button type="submit" disabled={mutation.isPending}>
        {mutation.isPending ? 'Saving...' : 'Create Post'}
      </button>
    </form>
  );
};

4. Real-Time Scenarios & Usages

  • Submitting a login form, creating a new task in a Todo list, or deleting an image from a gallery.
  • Disabling a submit button automatically while mutation.isPending is true to prevent double-submissions.

5. Advantages & Disadvantages

Advantages: Provides a standardized API for handling loading/error states during form submissions, completely replacing messy try/catch blocks in UI components.
Disadvantages: Doesn't automatically update the UI after a successful save. You must manually trigger a cache invalidation (see next topic).

⚡ Interview Corner

Q: What is "Optimistic Updates" using `useMutation`?
A: An optimistic update is when you update the UI before the server responds. Inside onMutate, you manually manipulate the React Query cache to show the new item instantly (giving a snappy user experience). If the request fails (caught in onError), you automatically roll the cache back to its previous state.

5. Query Invalidation

1. Definition

Query Invalidation is the process of telling React Query that specific cached data is now outdated (stale). This forces React Query to automatically refetch that data in the background to update the UI.

2. Explanation

  • When you successfully create a new Post via useMutation, the cache for your useQuery(['posts']) is now outdated (it doesn't have the new post).
  • Instead of manually adding the item to the cache, you use queryClient.invalidateQueries({ queryKey: ['posts'] }).
  • React Query sees the cache is invalid, executes a background refetch for `['posts']`, and automatically updates your UI components with the fresh data.

3. Example Code

import { useMutation, useQueryClient } from '@tanstack/react-query';
import axios from 'axios';

const useCreatePost = () => {
  // 1. Get access to the global query client
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: (newPost) => axios.post('/api/posts', newPost),
    onSuccess: () => {
      // 2. Invalidate the 'posts' query to trigger a refetch
      queryClient.invalidateQueries({ queryKey: ['posts'] });
    }
  });
};

4. Real-Time Scenarios & Usages

  • Clicking a "Delete User" button in an admin table. On success, you invalidate the ['users'] table query so the deleted user instantly disappears from the grid.

5. Advantages & Disadvantages

Advantages: Keeps the UI perfectly synchronized with the backend with almost zero manual state management.
Disadvantages: Invalidation triggers a new network request. If the payload is massive, an "Optimistic Update" (manually updating cache without a refetch) might save bandwidth.

⚡ Interview Corner

Q: How does Query Invalidation matching work? Will invalidating `['users']` also invalidate `['users', 5]`?
A: Yes, by default, query invalidation uses "fuzzy matching". If you invalidate the key ['users'], React Query will invalidate any query key that starts with that array. This means ['users'], ['users', 5], and ['users', { status: 'active' }] will all be refetched automatically.

6. Integrating Custom Axios Client with React Query

1. Definition

This is the ultimate enterprise pattern. It combines the powerful HTTP interceptors and token management of Axios with the caching and UI synchronization capabilities of React Query.

2. Explanation

  • React Query is backend agnostic—it doesn't care *how* you fetch data, it just needs a Promise.
  • By passing an Axios Instance (from Module 7) into React Query's `queryFn`, you ensure that all fetches automatically utilize your BaseURLs, JWT token injections, and global 401 refresh flows.

3. Example Code

// src/api/axiosClient.js
import axios from 'axios';
export const apiClient = axios.create({ baseURL: 'https://api.com' });
// ... interceptor logic here ...

// src/hooks/useUsers.js
import { useQuery } from '@tanstack/react-query';
import { apiClient } from '../api/axiosClient';

// Clean, modular custom hook integrating both technologies
export const useUsers = () => {
  return useQuery({
    queryKey: ['users'],
    queryFn: async () => {
      // apiClient automatically injects JWT tokens
      const { data } = await apiClient.get('/users');
      return data; // React query caches this data
    }
  });
};

4. Real-Time Scenarios & Usages

  • This exact combination is used in almost all highly scalable, secure, modern React architectures (replacing the older Redux + Redux Thunk + Fetch architectures).

5. Advantages & Disadvantages

Advantages: Ultimate separation of concerns. Axios handles the low-level networking and security. React Query handles the high-level caching and component synchronization.
Disadvantages: Requires understanding two distinct library APIs and making sure error throwing in Axios properly rejects Promises so React Query registers the isError state.

⚡ Interview Corner

Q: If Axios catches a 500 server error in an interceptor, how does React Query know about it?
A: For React Query to know a request failed and trigger its isError state, the queryFn must throw an error (or return a rejected promise). Therefore, inside your Axios response interceptor, if an error occurs, you must absolutely return Promise.reject(error). If you just log it and don't reject it, React Query will think the request succeeded with empty data.
