Advanced Data Fetching & Axios

React Masterclass Module 7

← Back to Roadmap

📝 How to use this guide: Fetching data is the backbone of any dynamic React app. In this module, we transition from the basic built-in `fetch` API to using **Axios**, the industry standard for enterprise applications. Pay massive attention to **Interceptors and JWT Tokens**, as these are guaranteed interview topics for mid-to-senior frontend roles.

1. fetch API vs. axios

1. Definition

`fetch` is the native browser API for making HTTP requests. `axios` is a popular, third-party, promise-based HTTP client that provides a cleaner API and advanced features out of the box.

2. Explanation

  • JSON Parsing: With fetch, you must manually call .json() on the response. axios automatically transforms the response data into JSON.
  • Error Handling: fetch only rejects a promise on network failures. It will successfully resolve even for 404 or 500 errors. axios automatically rejects the promise for any HTTP error status (4xx, 5xx).
  • Timeouts & Cancellation: axios has built-in support for request timeouts and easily canceling requests, which requires cumbersome AbortControllers in fetch.

3. Example Code

// Using Fetch (More Boilerplate)
const getUserFetch = async () => {
  try {
    const res = await fetch('/api/user');
    if (!res.ok) throw new Error('HTTP status ' + res.status);
    const data = await res.json(); // Manual parsing
    console.log(data);
  } catch (err) { /* handle error */ }
};

// Using Axios (Cleaner)
import axios from 'axios';
const getUserAxios = async () => {
  try {
    const res = await axios.get('/api/user');
    console.log(res.data); // Auto-parsed!
  } catch (err) { /* Automatically catches 404s, 500s, etc. */ }
};

4. Real-Time Scenarios & Usages

  • fetch is perfectly fine for simple side-projects or lightweight components that make one or two generic API calls without needing authentication wrappers.
  • axios is heavily preferred in enterprise SPAs that require global error handling, file uploads with progress tracking, and complex token authentication.

5. Advantages & Disadvantages

Advantages (Axios): Less boilerplate, automatic JSON parsing, wider browser support, built-in XSRF protection.
Disadvantages (Axios): Adds an external dependency to your bundle size, whereas fetch is completely free and native to the browser.

⚡ Interview Corner

Q: Why does a `fetch` request not trigger the `catch` block on a 404 Not Found error?
A: The fetch API only considers a request "failed" (and rejects the promise) if it encounters a network error or is prevented from completing (like CORS issues or losing internet connection). An HTTP 404 or 500 is technically a successfully completed request from the browser's perspective—it successfully received the server's error response. You must manually check response.ok.

2. Setting up an Axios Client Instance

1. Definition

An Axios instance is a custom-configured copy of Axios that pre-defines settings like the Base URL, headers, and timeouts for a specific API, so you don't have to repeat them in every single request.

2. Explanation

  • Instead of importing the default axios object, you create an instance using axios.create() and export it from a centralized file.
  • It allows you to define the baseURL. When you make a request to /users, it automatically prepends https://api.mycompany.com/v1.
  • You can set default headers (like Content-Type: application/json) that apply to all requests made by that instance.

3. Example Code

// src/api/axiosClient.js
import axios from 'axios';

const apiClient = axios.create({
  baseURL: process.env.REACT_APP_API_URL || 'https://api.example.com/v1',
  timeout: 10000, // Abort request if it takes longer than 10 seconds
  headers: {
    'Content-Type': 'application/json',
    'Accept': 'application/json'
  }
});

export default apiClient;

// src/components/Users.jsx
// Usage: No need to specify the full URL or headers!
// const res = await apiClient.get('/users');

4. Real-Time Scenarios & Usages

  • Connecting to different microservices. You might create authClient for the authentication server and dataClient for the main database API, each with different base URLs and timeout settings.

5. Advantages & Disadvantages

Advantages: Follows the DRY (Don't Repeat Yourself) principle. Makes environment switching (Dev vs Prod URLs) effortless.
Disadvantages: None. This is a mandatory best practice for any React application using Axios.

⚡ Interview Corner

Q: How do you handle different API base URLs for Development, Staging, and Production?
A: We use Environment Variables (e.g., .env.development and .env.production). In React, we prefix these with REACT_APP_ (or VITE_ for Vite setups). We then assign the Axios baseURL to process.env.REACT_APP_API_URL. The build tool automatically injects the correct URL based on the environment being built.

3. Axios Interceptors (Request & Response)

1. Definition

Interceptors are middleware functions that React runs seamlessly right before an HTTP request is sent out (Request Interceptor), or right before the response is handed back to your `.then()` or `catch()` blocks (Response Interceptor).

2. Explanation

  • Request Interceptors: Used to modify the request config globally. The most common use case is pulling an Authentication Token from local storage and attaching it to the Authorization header before the request hits the network.
  • Response Interceptors: Used to handle global logic based on what the server sends back. You can globally format the data, or intercept global errors (like logging a user out immediately if the server returns a 401 Unauthorized).

3. Example Code

import apiClient from './axiosClient';

// --- REQUEST INTERCEPTOR ---
apiClient.interceptors.request.use(
  (config) => {
    // 1. Get token from storage
    const token = localStorage.getItem('accessToken');
    // 2. Attach to headers if it exists
    if (token) {
      config.headers.Authorization = `Bearer ${token}`;
    }
    return config; // Must return config to proceed
  },
  (error) => Promise.reject(error)
);

// --- RESPONSE INTERCEPTOR ---
apiClient.interceptors.response.use(
  (response) => {
    // We can simplify the response so components only get the raw data
    return response.data; 
  },
  (error) => {
    // Global error logging
    console.error("API Error intercepted!", error);
    return Promise.reject(error);
  }
);

4. Real-Time Scenarios & Usages

  • Attaching Bearer JWT tokens to every secure route.
  • Triggering a global "Loading Spinner" in Redux right before a request starts, and turning it off when the response arrives.
  • Centralized logging of API performance metrics.

5. Advantages & Disadvantages

Advantages: Completely abstracts repetitive logic (like token injection and error formatting) away from your React components, keeping UI components perfectly clean.
Disadvantages: Errors thrown inside an interceptor can be difficult to trace in the browser's network tab if not logged properly.

⚡ Interview Corner

Q: What happens if an interceptor function does not explicitly `return config` (request) or `return response` (response)?
A: The entire HTTP request lifecycle will hang and fail. Axios relies on the interceptor passing the modified configuration object down the chain. If it returns undefined, the request has no URL or headers to execute with, resulting in a crash.

4. Handling JWT Tokens (Auth & Refreshes)

1. Definition

JSON Web Tokens (JWT) are secure strings used for stateless authentication. In React, managing JWTs involves storing them, attaching them to requests, and implementing a "Token Refresh" flow to maintain security without kicking the user out.

2. Explanation

  • Storage: Access tokens (short-lived, ~15 mins) are often stored in memory or localStorage. Refresh tokens (long-lived, ~7 days) should ideally be stored in secure, HttpOnly cookies managed by the backend.
  • The 401 Flow: When an access token expires, the server returns a 401 Unauthorized status.
  • Instead of forcing the user to log in again, an Axios Response Interceptor catches the 401.
  • It pauses the original request, calls a /refresh-token endpoint to get a new access token, updates local storage, and then silently retries the original failed request with the new token.

3. Example Code

apiClient.interceptors.response.use(
  (response) => response,
  async (error) => {
    const originalRequest = error.config;

    // If error is 401 and we haven't already retried this request
    if (error.response?.status === 401 && !originalRequest._retry) {
      originalRequest._retry = true; // Prevent infinite loops

      try {
        // Attempt to get a new token from the backend
        const res = await axios.post('/api/auth/refresh');
        const newAccessToken = res.data.accessToken;
        
        // Save new token
        localStorage.setItem('accessToken', newAccessToken);
        
        // Update the failed request's header and retry it
        originalRequest.headers.Authorization = `Bearer ${newAccessToken}`;
        return apiClient(originalRequest); 
      } catch (refreshError) {
        // If refresh fails (refresh token expired), force logout
        localStorage.clear();
        window.location.href = '/login';
        return Promise.reject(refreshError);
      }
    }
    return Promise.reject(error);
  }
);

4. Real-Time Scenarios & Usages

  • Any modern SaaS application (banking, social media, enterprise dashboards) requires this pattern so users aren't abruptly logged out every 15 minutes while working.

5. Advantages & Disadvantages

Advantages: Provides airtight security (due to short-lived access tokens) while maintaining a seamless, uninterrupted user experience.
Disadvantages: Highly complex to implement properly. Edge cases occur if multiple API requests fail with 401 simultaneously (requires a queue system in the interceptor to hold requests while the token refreshes).

⚡ Interview Corner

Q: Why is storing JWTs in `localStorage` considered a security risk?
A: localStorage is vulnerable to Cross-Site Scripting (XSS) attacks. If an attacker manages to inject malicious JavaScript into your React app, that script can easily read localStorage.getItem('token') and steal the user's identity. The most secure architectural approach is for the backend to send the token inside an HttpOnly cookie, which JavaScript cannot read, completely mitigating XSS token theft.

5. Error Handling Best Practices

1. Definition

Proper error handling ensures that when an API request fails, the application doesn't crash, the UI accurately reflects the failure state, and the user receives an actionable, friendly message instead of a cryptic console error.

2. Explanation

  • Standardize Errors: Backends often send specific error messages inside the response body (e.g., err.response.data.message).
  • Global vs. Local: Fatal errors (like 500 Server Error or Network Offline) should be handled globally via Axios interceptors and trigger a global Toast notification.
  • Specific errors (like "Username already taken" - 400 Bad Request) should be caught locally inside the component's try/catch block to show inline red text beneath a specific input field.

3. Example Code

const submitForm = async (userData) => {
  try {
    const data = await apiClient.post('/users', userData);
    toast.success("User created!"); // Success UI
  } catch (error) {
    // 1. Check if we got a response from the server (e.g., 400 Validation Error)
    if (error.response) {
      const serverMsg = error.response.data.message;
      setErrorMessage(serverMsg || "Validation failed.");
    } 
    // 2. The request was made but no response was received (Network error/Timeout)
    else if (error.request) {
      setErrorMessage("Network error. Please check your connection.");
    } 
    // 3. Something happened in setting up the request that triggered an Error
    else {
      setErrorMessage("An unexpected error occurred.");
    }
  }
};

4. Real-Time Scenarios & Usages

  • Displaying inline form validation errors returned from the server API.
  • Redirecting users to a generic "500 Internal Server Error" page if a critical microservice goes down.

5. Advantages & Disadvantages

Advantages: Massive improvement in User Experience and trust. Prevents users from clicking "Submit" 10 times because they don't know the server failed.
Disadvantages: Requires tight coordination with backend developers to ensure they return a standardized, predictable error JSON format across all API endpoints.

⚡ Interview Corner

Q: What is the architectural difference between handling 4xx vs 5xx errors in the frontend?
A: 4xx errors (Client Errors like 400 Bad Request, 401 Unauthorized, 404 Not Found) imply the user did something wrong (bad password, missing fields). These should usually be handled locally within the component to give the user a chance to fix their input. 5xx errors (Server Errors like 500 Internal Server Error, 503 Service Unavailable) imply the backend is broken. The user cannot fix this. These should be handled globally, displaying a "Try again later" toast and logging the event to a monitoring service.
