The Hooks Ecosystem (In-Depth)

React Masterclass Module 4

← Back to Roadmap

📝 How to use this guide: This module is the beating heart of modern React. Hooks allow functional components to "hook into" React state and lifecycle features. Interviewers will grill you on these. Pay special attention to the Interview Corners regarding dependency arrays and referential equality!

1. useState

1. Definition

`useState` is a hook that allows you to add state variables to functional components. It returns an array with the current state value and a function to update it.

2. Explanation

  • It takes an initial state value as its only argument.
  • Updating state triggers a re-render of the component.
  • State updates in React are asynchronous. Calling the setter function does not immediately mutate the variable on the next line of code; it schedules an update for the next render.
  • If the new state depends on the previous state, you should pass a callback function to the setter.

3. Example Code

const Counter = () => {
  const [count, setCount] = useState(0);

  const handleIncrement = () => {
    // Functional update ensures we have the absolute latest state
    setCount(prevCount => prevCount + 1);
  };

  return <button onClick={handleIncrement}>{count}</button>;
};

4. Real-Time Scenarios & Usages

  • Tracking form input values (controlled components).
  • Toggling UI elements (e.g., Modals, Dropdowns).
  • Storing fetched API data locally before displaying it.

5. Advantages & Disadvantages

Advantages: Simple and intuitive API for managing local state.
Disadvantages: Not ideal for highly complex state objects with deeply nested logic (use useReducer instead).

⚡ Interview Corner

Q: What happens if you call `setCount(count + 1)` three times in a row inside a single synchronous function?
A: The count will only increase by 1. Because state updates are asynchronous and batched, all three calls look at the closure's original value of count (e.g., 0). They all schedule an update to set the state to 0 + 1. To fix this, use the functional update: setCount(prev => prev + 1).

2. useEffect

1. Definition

`useEffect` lets you synchronize a component with an external system. It allows you to perform side effects (data fetching, subscriptions, manually changing the DOM) inside functional components.

2. Explanation

  • It runs after the component has rendered to the screen.
  • It takes two arguments: a setup function and an optional dependency array.
  • No array: Runs after every render.
  • Empty array `[]`: Runs only once when the component mounts.
  • Array with values `[id]`: Runs when the component mounts, and re-runs if `id` changes between renders.
  • The setup function can return a cleanup function, which runs before the component unmounts or before the effect re-runs.

3. Example Code

const Timer = ({ isActive }) => {
  useEffect(() => {
    if (!isActive) return;
    
    // Setup
    const intervalId = setInterval(() => console.log('Tick'), 1000);
    
    // Cleanup function
    return () => {
      clearInterval(intervalId);
    };
  }, [isActive]); // Re-run effect if isActive changes
  
  return <div>Timer is running...</div>;
};

4. Real-Time Scenarios & Usages

  • Fetching data from a REST API on page load.
  • Setting up WebSocket connections or event listeners (like window.addEventListener('scroll')).

5. Advantages & Disadvantages

Advantages: Combines componentDidMount, componentDidUpdate, and componentWillUnmount into a single API.
Disadvantages: Missing dependencies in the array is the #1 cause of "stale closure" bugs. Adding too many dependencies can cause infinite rendering loops.

⚡ Interview Corner

Q: How do you cause an infinite loop with `useEffect`?
A: By omitting the dependency array (or putting a state variable in it) and then updating that exact same state variable inside the effect.
useEffect(() => { setCount(count + 1); }, [count]).
Render -> Effect runs -> State updates -> Re-render -> Effect runs -> Infinite loop.

3. useRef

1. Definition

`useRef` is a hook that lets you reference a value that’s not needed for rendering. Mutating a ref does NOT trigger a re-render.

2. Explanation

  • It returns an object with a single property: { current: initialValue }.
  • Unlike state, the value of ref.current persists across re-renders without causing a re-render itself.
  • It is primarily used as an "escape hatch" to directly access real DOM elements (bypassing the Virtual DOM).

3. Example Code

const FocusInput = () => {
  const inputRef = useRef(null);
  const rendersCount = useRef(0);

  const handleFocus = () => {
    // 1. Directly accessing the DOM node
    inputRef.current.focus();
    
    // 2. Mutating a value without causing a re-render
    rendersCount.current += 1; 
  };

  return (
    <>
      <input ref={inputRef} type="text" />
      <button onClick={handleFocus}>Focus Input</button>
    </>
  );
};

4. Real-Time Scenarios & Usages

  • Managing focus, text selection, or media playback (videoRef.current.play()).
  • Storing mutable interval IDs (from setInterval) so they can be cleared later without triggering re-renders.

5. Advantages & Disadvantages

Advantages: Safely stores data outside the render cycle. Provides a bridge to Vanilla JS DOM manipulation.
Disadvantages: Overusing it to manipulate the DOM manually defeats the purpose of React's declarative state-driven philosophy.

⚡ Interview Corner

Q: Why shouldn't you read or write `ref.current` during rendering?
A: React expects components to behave like pure functions. Reading or writing a mutable value (like a ref) during the render phase makes component behavior unpredictable and breaks concurrent rendering features. Refs should only be accessed in event handlers or inside useEffect.

4. useContext

1. Definition

`useContext` lets you read and subscribe to context from your component. Context provides a way to pass data through the component tree without having to pass props down manually at every level (prop drilling).

2. Explanation

  • You create a context using React.createContext().
  • You wrap a parent component in a <Context.Provider value={data}>.
  • Any nested child, no matter how deep, can call useContext(Context) to access data.
  • When the Provider's value changes, all components consuming that context will re-render.

3. Example Code

import { createContext, useContext } from 'react';

// 1. Create Context
const ThemeContext = createContext('light');

const App = () => {
  // 2. Provide Context
  return (
    <ThemeContext.Provider value="dark">
      <Layout /> {/* Deeply nested tree */}
    </ThemeContext.Provider>
  );
};

const DeeplyNestedButton = () => {
  // 3. Consume Context
  const theme = useContext(ThemeContext);
  return <button className={`btn-${theme}`}>Click Me</button>;
};

4. Real-Time Scenarios & Usages

  • Global Theme settings (Light/Dark mode).
  • Current authenticated user data (User Auth state).
  • Localization/Translation language settings.

5. Advantages & Disadvantages

Advantages: Completely solves the "prop drilling" problem for global data.
Disadvantages: Poor performance if used for high-frequency state updates. Every time the context value changes, every component using useContext re-renders, bypassing React.memo.

⚡ Interview Corner

Q: Why wouldn't you use Context to manage the state of a fast-typing text input globally?
A: Because Context lacks fine-grained reactivity. If you put rapid state (like keystrokes) in Context, every single component subscribed to that Context will re-render on every single keystroke, causing massive lag. Redux or Zustand are better suited for fast, granular global state updates.

5. useReducer

1. Definition

`useReducer` is an alternative to `useState` for managing complex state logic that involves multiple sub-values or when the next state depends heavily on the previous one. It uses a "reducer" function (like Redux).

2. Explanation

  • You provide a reducer function (state, action) => newState and an initial state.
  • It returns the current state and a dispatch function.
  • Instead of updating state directly (like setState(data)), you dispatch "actions" describing what happened (e.g., dispatch({ type: 'INCREMENT' })).
  • The reducer function contains a switch statement that calculates the new state based on the action type.

3. Example Code

const reducer = (state, action) => {
  switch (action.type) {
    case 'increment': return { count: state.count + 1 };
    case 'decrement': return { count: state.count - 1 };
    default: throw new Error();
  }
};

const Counter = () => {
  const [state, dispatch] = useReducer(reducer, { count: 0 });

  return (
    <>
      Count: {state.count}
      <button onClick={() => dispatch({ type: 'increment' })}>+</button>
    </>
  );
};

4. Real-Time Scenarios & Usages

  • Managing forms with many interconnected fields (where changing one field impacts another).
  • Handling data fetching states (loading, success, error) in a clean, predictable way instead of managing 3 separate useState hooks.

5. Advantages & Disadvantages

Advantages: Predictable state transitions. Centralizes complex state logic. Easier to test the reducer function because it is a pure JavaScript function independent of React.
Disadvantages: Requires more boilerplate code than useState.

⚡ Interview Corner

Q: When would you choose `useReducer` over `useState`?
A: I would choose useReducer when state is an object with multiple nested properties, when multiple state variables need to be updated together in tandem, or when the next state relies heavily on complex logic regarding the previous state. It decouples the "what happened" (dispatch) from "how the state updates" (reducer).

6. useMemo

1. Definition

`useMemo` is a hook that lets you cache (memoize) the result of a calculation between re-renders. It prevents expensive operations from running on every single render.

2. Explanation

  • It takes a calculation function and a dependency array.
  • React will only recalculate the value if one of the dependencies has changed since the last render.
  • If dependencies haven't changed, React skips the calculation and returns the cached value.
  • It is strictly a performance optimization tool.

3. Example Code

const DataGrid = ({ rawData, theme }) => {
  // Sorting/filtering a massive array is expensive.
  // We only recalculate this if 'rawData' changes.
  // Changing 'theme' triggers a re-render, but bypasses this calculation.
  const processedData = useMemo(() => {
    return rawData.filter(item => item.isActive).sort();
  }, [rawData]);

  return <div className={theme}>...</div>;
};

4. Real-Time Scenarios & Usages

  • Filtering or sorting large datasets or arrays.
  • Preserving the referential equality of an object or array passed down to an optimized child component.

5. Advantages & Disadvantages

Advantages: Saves CPU cycles by preventing unnecessary expensive calculations.
Disadvantages: Overusing it actually hurts performance. Caching values takes memory and checking dependencies takes time. Do not use it for simple math or basic string concatenations.

⚡ Interview Corner

Q: What is "Referential Equality" and how does `useMemo` help with it?
A: In JS, two identical objects are not equal ({} === {} is false) because they occupy different memory spaces. Every time a component renders, inline objects/arrays are recreated at new memory addresses. If you pass an array as a prop to a child, the child thinks the prop changed and re-renders. useMemo caches the object, maintaining its exact memory reference across renders, preventing the child from re-rendering.

7. useCallback

1. Definition

`useCallback` is a hook that caches a function definition between re-renders. While `useMemo` caches the result of a function, `useCallback` caches the function itself.

2. Explanation

  • Every time a functional component renders, any functions declared inside it are recreated in memory.
  • Usually, this is fast and fine. However, if you pass that function as a prop to a deeply nested, optimized child component (wrapped in React.memo), the child will re-render because it sees a "new" function reference.
  • useCallback locks the function's memory reference as long as its dependencies don't change.

3. Example Code

const Parent = () => {
  const [text, setText] = useState('');

  // Caching the function so its reference doesn't change on every keystroke
  const handleSubmit = useCallback(() => {
    api.submitData(text);
  }, [text]); // Re-create function only when 'text' changes

  return (
    <>
      <input value={text} onChange={e => setText(e.target.value)} />
      {/* OptimizedChild won't re-render uselessly because handleSubmit reference is stable */}
      <OptimizedChild onSubmit={handleSubmit} /> 
    </>
  );
};

4. Real-Time Scenarios & Usages

  • Passing event handlers (like onClick or onSubmit) to highly optimized custom UI components (like a complex chart or grid).
  • When a function is used as a dependency inside a useEffect block elsewhere.

5. Advantages & Disadvantages

Advantages: Prevents unnecessary re-renders of child components.
Disadvantages: Just like useMemo, it adds memory overhead. Wrapping every single function in useCallback makes code ugly and slower. Only use it when passing functions to memoized child components.

⚡ Interview Corner

Q: What is the exact relationship between `useMemo` and `useCallback`?
A: They are essentially the same under the hood. useCallback(fn, deps) is functionally equivalent to writing useMemo(() => fn, deps). `useCallback` is just syntactic sugar specifically meant for caching functions so you don't have to write a function that returns another function.

8. useLayoutEffect

1. Definition

`useLayoutEffect` is identical to `useEffect` in signature, but it fires synchronously immediately after React performs all DOM mutations, but before the browser has a chance to paint those changes to the screen.

2. Explanation

  • useEffect is asynchronous. React renders -> Browser paints -> useEffect runs.
  • useLayoutEffect is synchronous. React renders -> useLayoutEffect runs (blocking the browser) -> Browser paints.
  • Because it blocks the browser from painting, it should only be used when absolutely necessary to prevent visual flickering.

3. Example Code

const Tooltip = ({ children }) => {
  const ref = useRef(null);
  const [tooltipHeight, setTooltipHeight] = useState(0);

  // Measure the DOM element BEFORE the user sees it, 
  // so we can adjust its position without a visible flicker/jump.
  useLayoutEffect(() => {
    setTooltipHeight(ref.current.getBoundingClientRect().height);
  }, []);

  return <div ref={ref} style={{ top: -tooltipHeight }}>{children}</div>;
};

4. Real-Time Scenarios & Usages

  • Measuring DOM elements (getting scroll height, element widths) to dynamically calculate layouts.
  • Preventing visual flickers (e.g., rendering an element, measuring it, and moving it to the right place before the user even sees it render in the wrong place).

5. Advantages & Disadvantages

Advantages: Perfect for solving UI flickering issues related to layout dimensions.
Disadvantages: It blocks visual updates. If you do heavy calculations inside it, your app will feel frozen and unresponsive. Default to useEffect 99% of the time.

⚡ Interview Corner

Q: Why does React give a warning if you use `useLayoutEffect` in a Server-Side Rendered (SSR) app like Next.js?
A: Because useLayoutEffect relies on reading the DOM (which only exists in the browser). On the server, there is no DOM. Therefore, the effect can't run on the server, causing a mismatch between what the server generates and what the client expects.

9. Custom Hooks

1. Definition

A Custom Hook is simply a JavaScript function whose name starts with "use" and that calls other React Hooks inside of it. It is the primary mechanism for extracting and reusing stateful logic across components.

2. Explanation

  • Custom hooks do NOT share state. They share stateful logic. Each component calling a custom hook gets its own completely independent state variables.
  • They must follow the Rules of Hooks (only call at the top level, only call from React functions).

3. Example Code

// 1. Defining the Custom Hook
function useWindowSize() {
  const [size, setSize] = useState({ width: window.innerWidth });

  useEffect(() => {
    const handleResize = () => setSize({ width: window.innerWidth });
    window.addEventListener('resize', handleResize);
    return () => window.removeEventListener('resize', handleResize);
  }, []);

  return size; // Return the state
}

// 2. Using it in components
const Navbar = () => {
  const { width } = useWindowSize(); // Clean, reusable logic!
  return <div>{width < 768 ? 'Mobile Nav' : 'Desktop Nav'}</div>;
};

4. Real-Time Scenarios & Usages

  • useFetch: Reusable API fetching logic with loading/error states.
  • useDebounce: Delaying rapid API calls during search input typing.
  • useLocalStorage: Syncing React state with the browser's local storage automatically.

5. Advantages & Disadvantages

Advantages: Drastically reduces code duplication. Keeps component files incredibly clean (separation of logic from UI).
Disadvantages: Poorly named or overly complex custom hooks can obscure what the code is actually doing.

⚡ Interview Corner

Q: What are the two absolute "Rules of Hooks"?
A: 1. Only call Hooks at the top level. Never call them inside loops, conditions, or nested functions. This ensures hooks are called in the exact same order every render (how React maps state internally).
2. Only call Hooks from React function components or other custom Hooks (not regular JS functions).

10. React 18+ Hooks (useTransition, useId)

1. Definition

React 18 introduced new hooks leveraging the Concurrent rendering engine. `useTransition` allows marking updates as non-urgent, and `useId` generates unique IDs for accessibility.

2. Explanation

  • `useTransition`: Returns [isPending, startTransition]. State updates wrapped in startTransition are marked as low-priority. If a high-priority update (like typing) occurs, React will pause the low-priority render.
  • `useId`: Generates a unique, stable string ID that works flawlessly across Server-Side Rendering (SSR) and Client-Side rendering.

3. Example Code

import { useTransition, useId } from 'react';

const Search = () => {
  const [query, setQuery] = useState('');
  const [isPending, startTransition] = useTransition();
  const id = useId(); // Generates e.g., ":r3:"

  const handleChange = (e) => {
    // High Priority: Update the input instantly
    setQuery(e.target.value); 
    
    // Low Priority: Filtering the massive list can wait if user keeps typing
    startTransition(() => {
      filterMassiveList(e.target.value);
    });
  };

  return (
    <>
      <label htmlFor={id}>Search:</label>
      <input id={id} onChange={handleChange} />
      {isPending ? <Spinner/> : <List />}
    </>
  );
};

4. Real-Time Scenarios & Usages

  • useTransition: Building highly responsive search bars that filter thousands of items locally without freezing the keyboard input.
  • useId: Connecting <label htmlFor> to <input id> in reusable UI components without hardcoding IDs or generating random numbers that break SSR.

5. Advantages & Disadvantages

Advantages: Massive UX improvements by keeping the main thread free for urgent interactions.
Disadvantages: Adds complexity to state flow. Overusing concurrent features when not needed can make debugging UI states difficult.

⚡ Interview Corner

Q: What is the difference between `useTransition` and `useDeferredValue`?
A: They solve the same problem (concurrent rendering) but are used differently. You use useTransition when you have access to the state-updating function (the setter). You use useDeferredValue when you only receive a value from a parent via props and want to defer rendering based on that value, without controlling the state itself.
