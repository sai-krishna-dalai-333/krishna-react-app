Global Client State (Redux Toolkit)

React Masterclass Module 9

← Back to Roadmap

📝 How to use this guide: While React Query handles Server State (API data), Redux Toolkit (RTK) is the industry standard for managing complex Client State (UI toggles, multi-step forms, global audio players). Interviewers love to ask about the differences between Context API and Redux, as well as how RTK solves traditional Redux boilerplate issues.

1. useContext vs. Redux (When to use which?)

1. Definition

`useContext` is a built-in React hook for Dependency Injection to avoid prop-drilling. Redux is an external state management architecture (a strict state machine) designed for highly predictable state updates.

2. Explanation

  • Context API is a way to pass data deeply into the tree. By default, it has no concept of "fine-grained reactivity"—if the context value changes, every component reading that context will re-render, even if they only needed a small piece of it.
  • Redux sits completely outside the React tree. Components "subscribe" to very specific slices of data. If `state.theme` updates, only components subscribed to `state.theme` re-render; components subscribed to `state.user` do not.

3. Example Code

// 1. Context (Good for static/slow-changing data)
const ThemeContext = createContext();
// Triggers re-renders across the entire app when theme changes
<ThemeContext.Provider value={theme}>...</ThemeContext.Provider>

// 2. Redux (Good for fast/frequent state changes)
// Component only re-renders if 'count' changes, ignoring other state
const count = useSelector(state => state.counter.count);

4. Real-Time Scenarios & Usages

  • Use Context: Theme settings (Dark/Light), User Auth status (Logged In/Out), Localization settings.
  • Use Redux: Complex multi-step checkout forms, global music/video player states, canvas dragging/dropping coordinates (fast-moving UI data).

5. Advantages & Disadvantages

Advantages (Redux): Performance optimization at scale. Time-travel debugging.
Disadvantages (Redux): Requires significantly more setup, boilerplate, and mental overhead compared to a simple Context Provider.

⚡ Interview Corner

Q: "Context API replaces Redux." Is this statement true or false?
A: False. They solve different problems. Context solves prop-drilling. Redux solves state management and performance. Context is not a state manager; it is just a transport mechanism. You still have to manage the state yourself using useState or useReducer at the top level before passing it into Context.

2. Redux Core Concepts vs. Redux Toolkit (RTK)

1. Definition

Redux is the architectural pattern (Store, Actions, Reducers). Redux Toolkit (RTK) is the official, opinionated toolset designed by the Redux team to make writing Redux logic easier and less verbose.

2. Explanation

  • Store: The global Javascript object holding all the state.
  • Actions: Objects describing what happened (e.g., { type: 'counter/increment' }).
  • Reducers: Pure functions that take the current state and an action, and return a completely new state object.
  • The RTK Difference: Legacy Redux required manually creating action types, action creators, and writing massive switch statements. RTK automates all of this. It also includes Immer.js under the hood, allowing you to write "mutating" logic safely.

3. Example Code

// Legacy Redux (Do NOT do this anymore)
const counterReducer = (state = { value: 0 }, action) => {
  switch (action.type) {
    case 'INCREMENT':
      // Manual immutable copying
      return { ...state, value: state.value + 1 }; 
    default:
      return state;
  }
};

// RTK Way (Under the hood, Immer makes this safe and immutable)
const counterSlice = createSlice({
  name: 'counter',
  initialState: { value: 0 },
  reducers: {
    increment: (state) => {
      state.value += 1; // Looks like a mutation, but it's safe!
    }
  }
});

4. Real-Time Scenarios & Usages

  • RTK is now the absolute standard. If you start a new React project and need Redux, you use @reduxjs/toolkit.

5. Advantages & Disadvantages

Advantages (RTK): Cuts boilerplate by 70%. Eliminates accidental state mutations (the #1 cause of Redux bugs).
Disadvantages: "Magic" abstraction. Juniors might learn RTK without understanding the core Redux unidirectional data flow principles underneath.

⚡ Interview Corner

Q: Why are Reducers required to be "Pure Functions"?
A: A pure function must always return the exact same output for the same input, and it must have absolutely no side effects (no API calls, no mutating external variables). Redux relies on pure functions because it determines if the UI needs to re-render by doing a shallow strict equality check (oldState === newState). If you mutate the old state, the reference stays the same, and React will completely fail to re-render.

3. Setting up an RTK Store (configureStore)

1. Definition

`configureStore` is the RTK API used to create the central Redux store. It automatically wraps standard Redux `createStore` and sets up developer tools and standard middlewares.

2. Explanation

  • It accepts a single configuration object.
  • You pass all your different slice reducers into the reducer property, which automatically combines them (acting like the old combineReducers).
  • It automatically enables the Redux DevTools browser extension.
  • It automatically adds redux-thunk middleware to allow asynchronous actions.

3. Example Code

// src/store/index.js
import { configureStore } from '@reduxjs/toolkit';
import cartReducer from '../features/cart/cartSlice';
import userReducer from '../features/user/userSlice';

export const store = configureStore({
  reducer: {
    cart: cartReducer, // state.cart
    user: userReducer, // state.user
  },
});

// src/index.js
import { Provider } from 'react-redux';
import { store } from './store';

root.render(
  <Provider store={store}>
    <App />
  </Provider>
);

4. Real-Time Scenarios & Usages

  • This is the absolute first step you take when setting up Redux in a brand new enterprise React application.

5. Advantages & Disadvantages

Advantages: Turns a previously 20-line, highly complex setup file (importing thunk, applying middleware, composing devtools) into a 5-line clean object.
Disadvantages: None. It is the definitive way to start Redux.

⚡ Interview Corner

Q: What is Middleware in the context of Redux?
A: Middleware provides a third-party extension point between dispatching an action and the moment it reaches the reducer. It is used for logging, crash reporting, routing, and especially for handling asynchronous API requests (like Redux Thunk or Saga) before allowing the final data to hit the synchronous reducers.

4. Creating Slices (createSlice)

1. Definition

`createSlice` is a function that accepts an initial state, an object of reducer functions, and a "slice name", and automatically generates action creators and action types that correspond to the reducers and state.

2. Explanation

  • In legacy Redux, you had three separate files: actions.js, constants.js, and reducer.js. createSlice combines these all into one standard "feature slice" file.
  • The name string is used to prefix the generated action types (e.g., name 'cart' + reducer 'addItem' = action type cart/addItem).
  • You export the generated actions for the components to use, and export the reducer for the configureStore to use.

3. Example Code

import { createSlice } from '@reduxjs/toolkit';

const cartSlice = createSlice({
  name: 'cart',
  initialState: { items: [], total: 0 },
  reducers: {
    // Action Creator is auto-generated based on this key name
    addItem: (state, action) => {
      // action.payload contains the data passed in from the component
      state.items.push(action.payload);
      state.total += action.payload.price;
    },
    clearCart: (state) => {
      state.items = [];
      state.total = 0;
    }
  }
});

// Export the Actions for components to dispatch
export const { addItem, clearCart } = cartSlice.actions;

// Export the Reducer for the Store
export default cartSlice.reducer;

4. Real-Time Scenarios & Usages

  • Managing complex, localized domains of your app (e.g., authSlice for user session, videoPlayerSlice for volume, playback status, and current time).

5. Advantages & Disadvantages

Advantages: The "Ducks Pattern" in one function. Massively improves developer velocity and code readability.
Disadvantages: Slices can become massive if a feature has too much state. It's important to break them down logically.

⚡ Interview Corner

Q: Under the hood, how does `createSlice` allow you to write `state.items.push()` without breaking Redux's strict immutability rules?
A: It uses a library called Immer. Immer wraps the state in a JavaScript Proxy object. It watches you "mutate" this proxy (like calling .push()), tracks all those changes, and then behind the scenes, securely generates a brand-new immutable copy of the state with those changes applied. You write mutating code; Redux gets immutable data.

5. Consuming State (useSelector) & Dispatching (useDispatch)

1. Definition

These are the two primary React-Redux hooks. `useSelector` reads data from the store and subscribes to updates. `useDispatch` returns a function that lets you send actions to the store to trigger state changes.

2. Explanation

  • useSelector takes a selector function (e.g., state => state.cart.total). It ensures the component only re-renders when that specific piece of state changes.
  • useDispatch gives you the dispatch function. You must pass an Action Creator (imported from your slice) into it to update the store.

3. Example Code

import { useSelector, useDispatch } from 'react-redux';
import { addItem } from '../features/cart/cartSlice';

const ProductCard = ({ product }) => {
  // 1. Read the state (Re-renders if 'total' changes)
  const cartTotal = useSelector((state) => state.cart.total);
  
  // 2. Get the dispatch trigger
  const dispatch = useDispatch();

  const handleBuy = () => {
    // 3. Dispatch the action with a payload
    dispatch(addItem(product));
  };

  return (
    <div>
      <h3>{product.name}</h3>
      <button onClick={handleBuy}>Add to Cart (Current Total: ${cartTotal})</button>
    </div>
  );
};

4. Real-Time Scenarios & Usages

  • Used inside literally every single React component that needs to interact with Redux. Replaces the legacy connect() Higher-Order Component and mapStateToProps.

5. Advantages & Disadvantages

Advantages: Hooks are incredibly clean and easy to compose compared to the old connect() wrappers.
Disadvantages: If you write a bad selector (e.g., returning a brand new object every time like useSelector(state => ({...state.cart}))), you will cause infinite re-render loops.

⚡ Interview Corner

Q: Why should you extract the smallest piece of state possible with `useSelector` instead of grabbing the whole object?
A: useSelector uses strict equality (===) to determine if it should force a re-render. If you do const cart = useSelector(state => state.cart), your component will re-render anytime anything in the cart changes. If you only need the item count, writing useSelector(state => state.cart.items.length) guarantees the component only re-renders if the length changes, saving massive amounts of performance.

6. Asynchronous Redux (createAsyncThunk)

1. Definition

`createAsyncThunk` is an RTK API that handles asynchronous lifecycles. It accepts an action type string and a promise-returning function (like an API call), and generates `pending`, `fulfilled`, and `rejected` action types automatically.

2. Explanation

  • Because Reducers must be synchronous pure functions, you cannot put an API `fetch` inside a slice reducer.
  • You create a Thunk outside the slice. The Thunk handles the API call.
  • Inside the slice, you use extraReducers to "listen" for the Thunk's lifecycle stages and update the state accordingly (e.g., setting isLoading = true on pending, and saving the data on fulfilled).

3. Example Code

import { createAsyncThunk, createSlice } from '@reduxjs/toolkit';
import axios from 'axios';

// 1. Create the Async Thunk
export const fetchUser = createAsyncThunk('user/fetchUser', async (userId) => {
  const response = await axios.get(`/api/users/${userId}`);
  return response.data; // This becomes action.payload in fulfilled
});

// 2. Handle the lifecycles in the Slice
const userSlice = createSlice({
  name: 'user',
  initialState: { data: null, status: 'idle' },
  reducers: {},
  extraReducers: (builder) => {
    builder
      .addCase(fetchUser.pending, (state) => {
        state.status = 'loading';
      })
      .addCase(fetchUser.fulfilled, (state, action) => {
        state.status = 'succeeded';
        state.data = action.payload;
      })
      .addCase(fetchUser.rejected, (state) => {
        state.status = 'failed';
      });
  },
});

4. Real-Time Scenarios & Usages

  • Traditionally used for absolutely all API fetching in Redux applications before React Query became popular.
  • Currently best used for complex async flows that aren't strictly API fetches (like complex IndexedDB interactions, or multi-step async file parsing in the browser).

5. Advantages & Disadvantages

Advantages: Standardizes async behavior in Redux. Removes the need to manually write and dispatch 3 different actions for every API call.
Disadvantages: Still highly verbose compared to React Query. Doesn't offer built-in cache invalidation or automatic refetching.

⚡ Interview Corner

Q: With the rise of React Query (TanStack Query), is Redux Thunk obsolete?
A: For data fetching, yes, React Query/RTK Query is vastly superior and has mostly made Thunks obsolete. However, Redux Thunk is still valuable for triggering complex, multi-step asynchronous business logic where the primary goal is updating various different client UI states sequentially, rather than just pulling data from a database.
